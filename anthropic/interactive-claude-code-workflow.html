🚀 Claude Code: Complete Learning Guide
Understanding AI-powered coding from the ground up

What is Claude Code?

Claude Code is an AI coding assistant that works through conversation. Instead of writing code yourself or copying snippets, you describe what you want in plain English, and Claude reads your project, proposes changes, and implements them—all while keeping you in control.

Why Claude Code is Special

🧠 Understands Your Entire Project

Claude doesn't just edit one file—it reads your whole codebase, understands relationships between files, and makes changes that fit your existing patterns.

💬 Conversational, Not Automated

You have a back-and-forth dialogue. Claude proposes what it will do, you approve or modify, and changes only happen with your consent.

📋 Respects Your Rules

Through a CLAUDE.md file, you tell Claude your project's conventions, and it follows them automatically—using your preferred frameworks, style guides, and patterns.

🔧 Handles Multiple Files

Add a feature that needs changes across 5 files? Claude coordinates all edits, maintains consistency, and even writes tests if you've told it to.

Claude Code vs. Other AI Tools

Tool How It Works Best For
GitHub Copilot Autocompletes code as you type Writing individual functions quickly
ChatGPT / Claude.ai Answers questions, explains code Learning concepts, getting explanations
Claude Code Edits your actual project files through conversation Building features, refactoring, multi-file changes

When to Use Claude Code

  • Adding new features: "Add user authentication with password reset"
  • Debugging: "The form isn't submitting—can you find and fix the issue?"
  • Refactoring: "Convert this class component to a functional component with hooks"
  • Understanding code: "Explain how the payment processing works in this project"
  • Writing tests: "Add unit tests for the UserService class"
⚠️ What Claude Code is NOT:
  • A "build my entire app" button (it's collaborative, not magical)
  • Fully autonomous (you review and approve everything)
  • A replacement for learning to code (it's a teaching tool too!)

How It Works (High-Level)

The Simple Version:

  1. You: Open terminal in your project folder
  2. You: Type claude "what you want"
  3. Claude: Reads your files and proposes a plan
  4. You: Review and approve (or ask for changes)
  5. Claude: Makes the edits
  6. You: Test the changes
  7. Claude: Offers to commit to Git
  8. You: Approve or reject the commit
✓ The Result: You get working, tested code that follows your project's conventions, without writing it all yourself. You stay in control, learn from watching Claude work, and move faster than coding alone.

✓ Specific Requests

Clear, specific, actionable

Step 2 & 3: Claude Reads & Proposes

This is where Claude's intelligence shines. It doesn't just look at one file—it understands your entire project context.

💡 What Claude Analyzes:
  • File structure and architecture patterns
  • Existing code style and conventions
  • Dependencies between files
  • Your CLAUDE.md rules (if present)
  • Recent changes and git history
  • Test files and documentation

Decision Points: The Heart of Control

The yellow diamonds are where you maintain control. Here's what happens at each:

Decision Point 1: Approve the Plan

What you're reviewing: The proposed approach, which files will change, what will be added/modified

Your options:

  • "yes" - Proceed as proposed
  • "actually, [different approach]" - Modify the plan
  • "no, let me think about this" - Stop and reconsider
  • "can you explain why you're doing X?" - Ask for clarification

Decision Point 2: Approve the Changes

What you're reviewing: The actual code changes, diffs, commit message

Best practice: Always test before committing!

  • Run the application locally
  • Check that new features work
  • Verify no old features broke
  • Review the code quality

The Power of CLAUDE.md

This is THE SECRET to getting consistent, high-quality results from Claude Code. Think of it as your project's constitution—rules Claude must follow.

Example CLAUDE.md for a Hospital Management System

# Hospital Management System

## Project Overview
HIPAA-compliant patient records system
Stack: React 18, Node.js 20, PostgreSQL, TypeScript

## Code Style Rules
- **Language**: TypeScript for ALL new code
- **Functions**: Must have JSDoc comments
- **Async**: Use async/await, NOT .then()
- **Logging**: NO console.log in production
- **Naming**: camelCase for functions, PascalCase for components

## File Structure
/src
  /components - React components (functional only)
  /hooks - Custom React hooks
  /api - API endpoint handlers
  /models - Database schemas (Prisma)
  /utils - Pure utility functions
  /tests - Test files (required for all features)

## Testing Requirements
- Every new feature MUST have tests
- Run `npm test` before committing
- Minimum 80% code coverage
- Use React Testing Library for components

## Security Rules
- All patient data encrypted at rest
- Never log PHI (Protected Health Information)
- All API calls require authentication
- Use environment variables for secrets

## Git Workflow
- Branch naming: `feature/description` or `fix/description`
- Commit messages: Imperative mood ("Add", not "Added")
- Create PRs from feature branches to `develop`
- Never commit directly to `main`

## API Conventions
- RESTful endpoints
- Use proper HTTP status codes
- Return errors in consistent format:
  ```json
  {
    "error": "Error message",
    "code": "ERROR_CODE"
  }
  ```

## Database
- Use Prisma migrations for schema changes
- Never do raw SQL queries
- All queries must use parameterization
- Add indexes for frequently queried fields

## Don't Do This
- Don't use `any` type in TypeScript
- Don't bypass authentication checks
- Don't store passwords in plain text
- Don't use inline styles in React
✓ With this CLAUDE.md, Claude will automatically:
  • Write in TypeScript, not JavaScript
  • Add JSDoc comments to every function
  • Create test files alongside features
  • Follow your naming conventions
  • Avoid security mistakes
  • Use your project's error format
  • Never use prohibited patterns

Common Pitfalls & How to Avoid Them

Pitfall #1: "Claude broke my code!"

What happened: You approved changes without testing, or without understanding what changed.

Solution:

  • Always ask "can you show me a diff?" before committing
  • Test the changes locally first
  • Ask Claude to explain what changed and why
  • Use git to revert if needed: git reset --hard HEAD~1

Pitfall #2: "Claude isn't following my style"

What happened: You don't have a CLAUDE.md file, so Claude guesses.

Solution:

  • Create a CLAUDE.md in your project root
  • Document your conventions clearly
  • Be specific: "Use Prettier with 2-space tabs" not "use good formatting"

Pitfall #3: "Claude made changes I didn't ask for"

What happened: Claude made reasonable but unintended changes.

Solution:

  • Review the proposed plan carefully at Decision Point 1
  • Ask: "Why are you changing file X?"
  • Be explicit: "Only modify the login form, don't touch anything else"

Pitfall #4: "I don't understand what Claude did"

What happened: The changes are complex and you need clarification.

Solution:

  • Ask: claude "explain the changes you just made"
  • Ask: claude "why did you use approach X instead of Y?"
  • Claude is a teaching tool—use it to learn!

Advanced Tips for Power Users

Tip 1: Chain Multiple Tasks

claude "add dark mode, then add a settings page where users can toggle it, then persist the preference in localStorage"

Claude will break this into steps and execute sequentially.

Tip 2: Ask for Alternatives

claude "what are three different ways to implement user authentication?"

Then choose the approach you prefer before implementing.

Tip 3: Use Claude for Code Review

Tip 4: Document as You Go

Keep your code documented automatically.

Terminal Basics

Quick prerequisite: Claude Code runs in the terminal. If you've never used a terminal before, this 5-minute crash course will teach you everything you need.

What is a Terminal?

A terminal is a text-based interface where you type commands instead of clicking. Think of it as having a conversation with your computer using precise instructions.

💡 Why use a terminal?
Terminals are powerful for development because they allow you to automate tasks, access advanced features, and use tools like Claude Code.

How to Open a Terminal

🪟 Windows

Press Win + R, type cmd, press Enter

Or search "Command Prompt" in Start Menu

🍎 macOS

Press Cmd + Space, type terminal, press Enter

Or find "Terminal" in Applications → Utilities

🐧 Linux

Press Ctrl + Alt + T

Or search "Terminal" in applications

Essential Commands (Only 3!)

1. Where am I?

$ pwd
/Users/yourname/projects

pwd = "print working directory" - shows your current location

2. What's in this folder?

$ ls
Documents Desktop my-project

ls (Mac/Linux) or dir (Windows) - lists files and folders

3. Move to a different folder

$ cd my-project
~/my-project$

cd = "change directory" - moves you into a folder

⚠️ Most Important Concept:
Commands run in your current location. Before running claude "...", use pwd to confirm you're in your project folder!

Quick Cheat Sheet

What You Want Command
See where you are pwd
List files here ls (Mac/Linux) or dir (Windows)
Go into a folder cd folder-name
Go up one level cd ..
Go to home folder cd ~
✓ That's it! You now know enough terminal basics to use Claude Code. Just remember: pwd to check location, cd to move around, ls to see what's there.

Folder Navigation

Why this matters: Claude Code works on files in your current directory. If you're in the wrong folder, it won't see your project files!

~/Documents/Projects/my-website$ pwd

⚠️ Common Mistake:

Windows

macOS

  1. Open Finder
  2. Navigate to project folder
  3. Right-click folder
  4. Select "New Terminal at Folder"
